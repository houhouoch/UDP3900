# 🚀 DMA2D 硬件加速与显存填充实验笔记

在完成 LTDC 基础配置后，本项目引入了 **DMA2D** 硬件加速引擎。通过实现高效的颜色填充（R2M）与内存搬运（M2M），验证了 SDRAM 显存映射的准确性以及“左切割屏幕”的物理显示方向。

---

## 🛠️ 1. 驱动底层架构实现

驱动程序针对 **RGB565** 格式实现了两种核心 DMA2D 操作模式，显著降低了 CPU 在图形渲染中的负载。

### A. R2M (Register-to-Memory) 模式
*   **用途**：极速纯色填充（如清屏、绘制矩形）。
*   **实现逻辑**：直接将颜色寄存器中的数值写入目标显存地址。
*   **线性地址计算**：基于屏幕物理宽度 240 像素进行偏移定位。
```text
Addr = Base_Addr + 2 * (240 * sy + sx)
```
    *(其中 `Base_Addr` 为 SDRAM 起始地址 `0xD0000000`)*

### B. M2M (Memory-to-Memory) 模式
*   **用途**：图像缓冲区高速搬运（如 LVGL 帧刷新、UI 贴图）。
*   **实现逻辑**：将源缓冲区（SRAM/QSPI）数据无感搬运至目标显存。
*   **Cache 一致性管理**：针对 Cortex-M7 内核，在启动 DMA2D 搬运前必须调用 `SCB_CleanDCache_by_Addr`，确保硬件读取到的是内存中的最新数据，防止画面出现条纹或旧数据残留。

---

## ⚡ 2. 硬件同步与中断机制

为了提高系统并行效率并确保操作的原子性，驱动采用了 **“中断驱动 + 信号同步”** 的逻辑。

1.  **全局状态量**：定义 `volatile uint8_t g_dma2d_done` 作为传输完成的原子标志。
2.  **中断处理 (ISR)**：在 `DMA2D_IRQHandler` 中捕获 **TCIF**（传输完成）中断，清除标志位后将 `g_dma2d_done` 置 1。
3.  **非阻塞同步**：
    ```c
    /* 启动 DMA2D 传输 */
    g_dma2d_done = 0;
    HAL_DMA2D_Start_IT(&hdma2d, source, destination, width, height);
    
    /* 等待硬件完成，确保时序同步 */
    while(g_dma2d_done == 0); 
    ```

## 🔍 3. 屏幕验证与方向测试

针对 $240 \times 960$ “左切割屏幕”的特殊物理特性，进行了以下回归测试：

### 🌈 方案一：标准 8 色彩条测试 (`LCD_Test_ColorBars`)
*   **目的**：验证 RGB565 颜色分量解析是否准确。
*   **操作**：将屏幕横向等分为 8 个区域，依次填充：**白、黄、青、绿、品、红、蓝、黑**。
*   **验证点**：确保红蓝通道无交换（Swap），确认显存起始地址对齐屏幕物理左上角。

### 📐 方案二：坐标系确定性测试
*   **操作**：调用 `ltdc_color_fill(0, 0, 99, 99, test_buffer)`。
*   **观察点**：在逻辑坐标 $(0, 0)$ 到 $(99, 99)$ 区域写入特定色块。
*   **结论**：通过观察小方块在长条形屏幕上的物理位置，最终确认了逻辑原点与扫描增量方向，为后续 LVGL UI 适配提供了准确的坐标映射表。

## ⚠️ 针对“左侧物理切割屏”的适配要点

由于物理屏幕分辨率为 $240 \times 960$，但在驱动 IC 逻辑中可能存在宽度冗余，配置 DMA2D 时需注意：

*   **分辨率设置**：   $240 \times 960$。
*   **行偏移 (Offline) 逻辑**：
    在 DMA2D 配置中，**OOR (Output Offset Register)** 必须严格设置为：
    `OOR = 屏幕总宽度 - 填充区域宽度` (即 $240 - width$)。
    > **原理**：这确保了 DMA2D 在完成当前行填充后，地址能够准确跳转到下一行的起始像素，防止画面出现斜向扭曲。

### 📖 开发者总结
通过 DMA2D 实验，我们不仅实现了性能上的飞跃，更重要的是摸清了切割屏的“物理脾气”。在 H750 @ 480MHz 的驱动下，利用 DMA2D 进行清屏操作几乎不占用 CPU 时间，这为后续复杂的 GUI 动画留出了充足的算力。
